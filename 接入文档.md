# beebook-parse 接入文档

对外提供 **书籍搜索** 与 **书籍正文解析** 两项能力，通过 HTTP JSON 对接。

---

## ⚠️ v2 关键变更说明（升级请先看）

> 旧版本 `POST /api/parse` 在文件本地不存在时会**同步阻塞最长 5 分钟**下载文件，客户端常见超时；v2 改为**真正异步**，行为有以下变化：

| 变化点 | v1（旧） | **v2（当前）** |
|---|---|---|
| **POST /api/parse 返回时机** | 同步等下载完成才返回 | **立刻返回 task_id**（≤ 100ms） |
| **轮询 status 取值** | `processing` / `completed` / `failed` | 新增 `pending` / `downloading` / `parsing`，**老值 `processing` 仍兼容**（等价于 `parsing`） |
| **新错误码 `429`** | 无 | 解析队列过载，**带 `Retry-After` 头**，上游应退避后重试 |
| **新错误码 `504`** | 无 | 单任务超时（默认 5 分钟），看门狗强制标记失败 |
| **服务重启** | 残留 `processing` 卡住 | 启动自愈一律清掉中间态,客户端重提即可(原书仍在 `books/` 窗口内无需重新下载) |

**升级 checklist**：
- [x] 客户端轮询代码:把 "看到 `processing` 继续轮询" 改成 "看到 `pending`/`downloading`/`parsing`/`processing` 都继续轮询"
- [x] 加 429 处理:读 `Retry-After` 头退避重试（建议加 jitter 防惊群）
- [x] 加 504 处理:任务超时，记 metric，可换 md5 重试
- [x] 提交 POST 后**不要再期待立刻拿到 completed**（除非命中缓存 `cached: true`）

## 0. 快速上手

三步完成一次"关键词 → 全文"端到端调用：

```bash
# 1) 搜索拿到 md5（推荐带格式+内容过滤，见 3.1.1）
curl -s "http://HOST:5555/api/search?q=中国哲学简史\
&ext=pdf&ext=epub&ext=fb2&ext=txt&content=book_nonfiction" \
  | jq '.data.results[0] | {md5, title, extension}'
# → {"md5":"2e80459214878bb9a530ac85e366c1a6","title":"中国哲学简史","extension":"epub"}

# 2) 提交解析（异步）
curl -s -X POST http://HOST:5555/api/parse \
     -H "Content-Type: application/json" \
     -d '{"md5":"2e80459214878bb9a530ac85e366c1a6","extension":"epub"}'
# → {"code":0,"data":{"task_id":"2e80..._default","status":"processing","cached":false}, ...}

# 3) 轮询结果
curl -s http://HOST:5555/api/parse/2e80459214878bb9a530ac85e366c1a6_default
# → {"code":0,"data":{"status":"completed","text":"...","total_length":182223, ...}, ...}
```

客户端统一用 `response.code === 0` 判断成功。

## 1. 通用约定

| 项 | 说明 |
|---|---|
| **Base URL** | `http://HOST:5555`（`HOST` 由部署方提供；默认端口可通过 `PORT` 环境变量改） |
| **协议** | HTTP/1.1，JSON body 必须 UTF-8，响应均为 UTF-8 JSON |
| **请求方法** | `GET` 查询类、`POST` 提交类 |
| **中文参数** | **必须 URL 编码**（详见下面搜索接口说明） |

### 1.1 统一响应结构

**所有**接口成功与失败都使用相同外壳：

```json
{
  "code": 0,
  "data": { "...业务数据..." },
  "message": "success",
  "timestamp": 1712345678900
}
```

| 字段 | 类型 | 含义 |
|---|---|---|
| `code` | int | `0` 表示成功；非 0 为业务错误码（见 1.2） |
| `data` | object / null | 业务载荷。错误时通常为 `null`，但某些错误（如 503）也会回传已有数据便于排查 |
| `message` | string | 提示信息。成功时为 `"success"`，错误时为中文错误描述 |
| `timestamp` | int | 响应生成时刻的毫秒级 Unix 时间戳 |

客户端**只需要判断 `code === 0`** 即可区分成功/失败，不必同时看 HTTP 状态码。
HTTP 状态码遵循 REST 语义（400/404/500/502 等），但**不是唯一判据**。

### 1.2 业务错误码

| code | 含义 | HTTP | 典型场景 | 上游建议 |
|---|---|---|---|---|
| `0` | 成功 | 200 | 正常返回 | — |
| `400` | 参数错误 | 400 | 缺 `q` / 缺 `md5` / body 非法 JSON | 修参数后重试 |
| `404` | 资源不存在 | 404 | `md5` 对应文件找不到 / `task_id` 不存在或过期 | 不要重试 |
| `429` | **过载限流** ⭐v2 | 429 | 解析队列已满 | **必须**读 `Retry-After` 头退避后重试 |
| `500` | 内部错误 | 500 | 未预期异常（应上报排查） | 上报 + 慎重重试 |
| `501` | 解析失败 | 200 | 解析器报错（不支持的格式、损坏文件等） | 换 md5 / engine |
| `502` | 上游不可用 | 502 | 所有 Anna's Archive 镜像都挂了 | 退避 30s 后重试 |
| `503` | 内容为空 | 200 | 解析完成但文本长度为 0（如纯扫描件未 OCR） | 换 md5 |
| `504` | **任务超时** ⭐v2 | 200 | 看门狗判定单任务超 5 分钟 | 换 md5 / 检查文件大小 |

### 1.3 URL 编码说明（透传相关）

**原则：服务端在能力范围内最大化兜底，调用方尽量不用管。**

#### ✅ 服务端已帮你做的

| 场景 | 服务端行为 |
|---|---|
| 中文、日文、韩文等非 ASCII 字符**未 URL 编码直接塞进 query** | 自动识别 UTF-8 字节在 Latin-1 下的 mojibake 形态并反解还原，透明修复 |
| 中文**已经 URL 编码** | 正常解析 |
| 大小写不敏感（`md5`、`ext`、`extension`） | 统一按小写处理 |
| 首尾空白、全角空格等 | `strip()` 自动清理 |

一句话：**中文字符调用方爱编不编都行**。

#### ⚠️ 仍需调用方编码的（HTTP 协议层限制，服务端收不到请求）

| 字符 | 为什么必须编码 |
|---|---|
| **空格** | URL 里空格非法，curl 会截断 URL，整个请求发不出去；浏览器会自动转成 `%20` |
| `&` `=` `?` `#` | URL 结构字符。不编码会被当成参数分隔、锚点等 |
| `+` `/` `%` | 保留字符，含义会被误解 |
| `"` `'` `<` `>` `{` `}` 等 | 各客户端行为不一致，部分会直接报错 |

这部分问题**发生在请求到达服务端之前**，无论服务端做什么兜底都救不了。

#### 🎯 推荐的三种安全做法（由易到难，随便挑一个）

**1️⃣ 用 HTTP 库（最推荐）**

HTTP 库都会自动 URL 编码，调用方一个字符都不用管。

```python
# Python requests —— params 里爱塞啥字符都行
requests.get(URL, params={
    "q": "python 编程 & AI",    # 空格、& 都能正确处理
    "ext": ["pdf", "epub"],
    "content": "book_nonfiction",
})
```

```javascript
// JS fetch + URLSearchParams
const p = new URLSearchParams({q: 'python 编程 & AI', content: 'book_nonfiction'});
['pdf','epub'].forEach(e => p.append('ext', e));
fetch(`${URL}?${p}`);
```

```go
// Go net/url
u, _ := url.Parse("http://HOST:5555/api/search")
q := u.Query()
q.Set("q", "python 编程 & AI")
q.Add("ext", "pdf")
q.Add("ext", "epub")
u.RawQuery = q.Encode()
http.Get(u.String())
```

**2️⃣ 命令行用 `curl -G --data-urlencode`**

```bash
curl -G "http://HOST:5555/api/search" \
  --data-urlencode "q=python 编程 & AI" \
  --data-urlencode "ext=pdf" \
  --data-urlencode "ext=epub" \
  --data-urlencode "content=book_nonfiction"
```

不用 `-G` 手写 URL 时**一定要手动编码空格 / 特殊字符**：

```bash
# ❌ 错误：空格会截断
curl "http://HOST:5555/api/search?q=python 编程"

# ✅ 正确：空格编码成 %20（中文可以不编码）
curl "http://HOST:5555/api/search?q=python%20编程"
```

**3️⃣ 前端 HTML 表单 / `<a href>`**

浏览器会自动编码表单字段和 href，调用方写原始文本即可：

```html
<a href="/api/search?q=python 编程">搜索</a>   <!-- 浏览器自动转 %20 -->
```

#### 🔍 结论一句话

> **用 SDK 调用 → 完全不用管编码。**
> **手写 URL 调用 → 中文可以不编码，但空格和 `& = ? # + / %` 等字符必须编码。**

## 2. 端点总览

| 方法 | 路径 | 用途 | 同步/异步 |
|---|---|---|---|
| GET  | `/api/search`           | 关键词搜索书籍（爬 Anna's Archive） | 同步（~1s） |

| GET  | `/api/formats`          | 查看支持的文件格式与解析引擎 | 同步 |
| GET  | `/health`               | 健康检查 | 同步 |

## 3. 接口详情

### 3.1 `GET /api/search` 书籍搜索

#### 请求参数

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `q` | string | ✅ | 搜索关键词。中文**需 URL 编码**，否则可能走成 mojibake 导致搜不到 |
| `lang` | string 或 string[] | ❌ | 语言过滤，可重复传。如 `?lang=zh` 或 `?lang=zh&lang=en` |
| `ext` | string 或 string[] | ❌ | 文件格式过滤，可重复传。如 `?ext=pdf` 或 `?ext=pdf&ext=epub&ext=mobi`。**强烈建议调用时传入 👉 3.1.1 建议白名单** |
| `content` | string 或 string[] | ❌ | 内容类型，可重复传。`book_nonfiction` / `book_fiction` / `book_unknown` / `journal_article` / `magazine` / `standards_document` / `comic` |
| `sort` | string | ❌ | 排序：`newest` / `oldest` / `largest` / `smallest` |
| `page` | int | ❌ | 页码，默认 `1`，每页固定 50 条 |

#### 请求示例

```bash
# 基本搜索
curl "http://HOST:5555/api/search?q=%E4%B8%AD%E5%9B%BD%E5%93%B2%E5%AD%A6%E7%AE%80%E5%8F%B2"

# 单格式过滤
curl "http://HOST:5555/api/search?q=python&lang=en&ext=pdf&sort=newest"

# ⭐ 推荐的标准调用（只要非小说类 + 4 种可解析格式，见 3.1.1）
curl "http://HOST:5555/api/search?q=python&ext=pdf&ext=epub&ext=fb2&ext=txt&content=book_nonfiction"

# 翻页
curl "http://HOST:5555/api/search?q=python&page=2"
```

```python
import requests
# 推荐：用 params 字典传 list，requests 会自动展开成多个 ?ext=
MAIN_EXTS = ["pdf", "epub", "fb2", "txt"]
r = requests.get("http://HOST:5555/api/search", params={
    "q": "中国哲学简史",
    "lang": "zh",
    "ext": MAIN_EXTS,
    "content": "book_nonfiction",
})
r.raise_for_status()
data = r.json()["data"]
for book in data["results"]:
    print(book["md5"], book["title"])
```

```javascript
// URLSearchParams 多值传法：传 [key, value] 对的数组
const MAIN_EXTS = ['pdf','epub','fb2','txt'];
const params = new URLSearchParams();
params.set('q', '中国哲学简史');
MAIN_EXTS.forEach(e => params.append('ext', e));
params.set('content', 'book_nonfiction');
const res = await fetch(`http://HOST:5555/api/search?${params}`);
const body = await res.json();
body.data.results.forEach(b => console.log(b.md5, b.title));
```

#### 3.1.1 ⭐ 强制推荐的搜索过滤组合

为保证召回质量，**所有对外搜索请求都应带上**以下过滤参数。服务端不做任何默认过滤，调用方必须自己传：

```
ext = pdf, epub, fb2, txt            （4 种主流可解析格式）
content = book_nonfiction            （只要非小说类图书）
```

##### ✅ 只保留这 4 种文件格式

| `ext` | 理由 | 本服务能否解析正文 |
|---|---|---|
| `pdf` | 量最大（AA 上 3 万+ 条）、扫描/排版书主流 | ✅ PyMuPDF / pdftotext |
| `epub` | 电子书首选格式，结构化好，体积小 | ✅ |
| `fb2` | 俄系电子书格式，部分中文老书/译本在用 | ✅ |
| `txt` | 纯文本，信息密度高、解析零损耗，6000+ 条 | ✅ |

##### ✅ 只保留 `book_nonfiction`（非小说类图书）

这是 Anna's Archive 内容类型的正选。**只要这一种**，其他类型全部被自动排除：

| 被排除的 content | 为什么排除 |
|---|---|
| `book_fiction`（小说） | 虚构类，信息/知识价值低 |
| `book_comic`（漫画） | 纯图片或 CBR/CBZ 归档，不是正文 |
| `magazine`（杂志） | 多栏排版、广告穿插、文字碎片，对"读书"场景无价值 |
| `journal_article`（期刊论文） | 偏学术短文，与通用阅读场景错配 |
| `standards_document`（标准文档） | 工具性 PDF，不是书 |
| `book_unknown`（未分类图书） | AA 没标类型，质量参差，宁可不要 |

##### ❌ 额外被 `ext` 白名单排除的格式

下表的格式不在 4 种白名单内，会被自动过滤：

| `ext` | 量级 | 排除原因 |
|---|---|---|
| `mobi` / `azw3` | 1100+ | Kindle 格式，读者分布更窄；epub 能覆盖大多数场景 |
| `djvu` | 712 | 学术扫描格式，读者少 |
| `docx` | 153 | 文档不是书 |
| `cbr` / `cbz` | 近 3000 | 漫画归档，不是书 |
| `doc` / `rtf` / `lit` / `chm` / `htm` / `html` / `htmlz` / `mht` / `odt` | 数十-数百 | 本服务不支持解析 |
| `ppt` / `xls` / `xlsx` | 约 200 | 演示文稿/表格，不是书 |
| `lrf` / `kfx` / `azw` / `pdb` / `prc` / `snb` / `updb` | 个位-几十 | 冷门电子书格式 |
| `zip` / `rar` / `7z` / `tar` | 数千 | 压缩包，不拆开不知道里面是啥 |
| `jpg` / `tif` / `json` | 少到 0 | 不是书或非正文数据 |

> 将来发现 mobi/azw3/djvu 其中某种对你的业务重要，可以单独加回 `ext` 白名单 —— 这部分由**调用方**决定。

##### 对 `total` 字段的影响

以"中国哲学简史"为例：

```
无过滤                                            → total=443
ext=pdf                                           → total=209
ext=pdf&ext=epub&ext=fb2&ext=txt                  → total≈300
content=book_nonfiction（不限 ext）               → total≈150
🔥 推荐组合（4 种 ext + book_nonfiction）         → total=93
```

召回从 443 压到 93，**精准度大幅提升**，当页 50 条基本都是真·非小说书籍。

#### 3.1.2 进阶：自定义 content 过滤（可选，多数业务不需要）

**正选**（`content=xxx`）：只保留指定类型

| 值 | 含义 |
|---|---|
| `book_nonfiction` | 非小说类图书（**推荐默认**） |
| `book_fiction` | 小说类图书 |
| `book_unknown` | 未分类图书 |
| `book_comic` | 漫画 |
| `magazine` | 杂志 |
| `journal_article` | 期刊论文 |
| `standards_document` | 标准文档 |

**反选**（`content=anti__xxx`，两个下划线）：排除指定类型，保留其余

```bash
# 只排除漫画（其他类型都保留）
curl "http://HOST:5555/api/search?q=python&content=anti__book_comic"

# 同时排除漫画和杂志
curl "http://HOST:5555/api/search?q=python\
&content=anti__book_comic&content=anti__magazine"
```

> ⚠️ **不要正选反选混用**。要么只传正选、要么只传 `anti__`，结果才可预期。

#### 字段说明

`data` 下的字段：

| 字段 | 说明 |
|---|---|
| `total` | **上游标注的命中总数**（可能远大于 `results.length`，因为每页只返 50 条） |
| `results` | 当页结果数组，**最多 50 条** |
| `parser` | 仅当走正则降级解析时才出现，值为 `"regex-fallback"`，正常不返回 |

| 结果字段 | 类型 | 说明 |
|---|---|---|
| `md5` | string | 文件 MD5，**是后续 `/api/parse` 的输入** |
| `title` | string | 书名 |
| `author` | string | 作者。原始格式，可能含多作者分隔符 `&`、`;` 等 |
| `publisher` | string | 出版社 + 年份（原样） |
| `extension` | string | 文件后缀小写：`pdf`/`epub`/`mobi`/`djvu`/... |
| `filesize_str` | string | 人类可读大小：`"21.5MB"`、`"0.5MB"` |
| `language` | string | 语言代码：`zh`、`en`... |

### 3.2 `POST /api/parse` 提交解析任务

根据 md5 **完全异步**解析一本书。**接口只做登记,立刻返回 `task_id`,后续状态靠 GET 轮询**。

服务端在后台依次执行(每一步都会更新 Redis meta 让客户端看到):

1. **`pending`** → 任务进队列,等待 worker 拾取
2. **`downloading`** → worker 从本地 `/app/books` 找文件;找不到调 Anna's Archive Fast Download API 下载
3. **`parsing`** → 按文件大小路由到小池(≤20MB,12 并发) / 大池(>20MB,4 并发);走对应解析器(PyMuPDF / EbookLib / ...)
4. **`completed`** → 结果**只**存 Redis(默认 3h TTL,到期自动失效,服务无状态)

整条链路有看门狗(默认 5 分钟超时)和过载背压(队列满返 429)兜底。

#### 请求

```bash
curl -X POST http://HOST:5555/api/parse \
     -H "Content-Type: application/json" \
     -d '{"md5":"2e80459214878bb9a530ac85e366c1a6","extension":"epub"}'
```

```python
import requests
r = requests.post("http://HOST:5555/api/parse",
                  json={"md5":"2e80459214878bb9a530ac85e366c1a6","extension":"epub"})
task = r.json()
```

#### 请求体字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `md5` | string | ✅ | 32 位小写 md5（来自 `/api/search` 结果的 `md5` 字段） |
| `extension` | string | ❌ | 文件后缀。省略时会从文件名自动识别，但显式传能避免 AA 下载时猜错后缀 |

#### 查询参数（可选）

| 参数 | 说明 |
|---|---|
| `engine` | 指定解析引擎。**一般不用传**，默认就是各格式的最佳引擎。仅 PDF 有多引擎可选：`pymupdf`（默认，推荐）/ `pdftotext`（备用，见下） |

**引擎选择说明（仅 PDF）**：

- **默认 `pymupdf`** —— 不传 `engine` 即用它。排版更干净、去重好、性能稳定，满足 99% 场景。
- **`pdftotext` 仅在以下场景考虑**：
  - 主流程拿到 `code=503`（pymupdf 抽出文本为空），可传 `?engine=pdftotext` 重试一次作为兜底
  - 需要极致召回率的 RAG 索引场景（文本量比 pymupdf 多 30%+，但含更多页眉页脚噪声）

#### 成功响应

**首次提交**(需要轮询):

```json
{

  "data": {
    "task_id": "2e80459214878bb9a530ac85e366c1a6_default",

    "cached": false
  },
  "message": "success",
  "timestamp": 1712345678900
```

**重复提交同一本书**(其他用户同时提交会合流到同一 task_id):

与首次提交响应完全相同,但 `status` 可能是 `pending`/`downloading`/`parsing` 中的当前状态。同 md5+engine 只会真正跑一次解析。

| 字段 | 说明 |

#### 推荐轮询策略 ⭐v2

```python
import time, requests

# v2 中间状态全集(包括兼容老值 processing)
PENDING_STATES = {"pending", "downloading", "parsing", "processing"}

class EmptyContent(Exception): pass
class ParseFailed(Exception): pass
class ParseTimeout(Exception): pass

def wait_for_text(task_id, timeout=600, interval=1.0):
    """
    timeout 建议给 600s(10 分钟),覆盖大文件下载+解析的最坏情况;
    服务端看门狗 5 分钟会强制把任务标 504,所以轮询不会真的等 10 分钟。
    """
    t_end = time.time() + timeout
    while time.time() < t_end:
        r = requests.get(f"http://HOST:5555/api/parse/{task_id}")
        body = r.json()
        code = body["code"]
        data = body.get("data") or {}
        status = data.get("status")

        # 成功且有文本
        if code == 0 and status == "completed":
            return data["text"]
        # 完成但文本为空 → 业务上相当于"这个 md5 用不了"
        if code == 503:
            raise EmptyContent(body["message"])
        # 解析失败(不支持格式、损坏文件等)
        if code == 501:
            raise ParseFailed(body["message"])
        # ⭐v2 任务超时(看门狗触发)
        if code == 504:
            raise ParseTimeout(body["message"])
        # ⭐v2 下载失败
        if code == 502:
            raise ParseFailed(f"下载失败: {body['message']}")
        # 404:任务不存在或过期
        if code == 404:
            raise KeyError(body["message"])
        # 任何中间态(pending/downloading/parsing/processing)继续轮询
        if code == 0 and status in PENDING_STATES:
            time.sleep(interval)
            continue
        # 兜底:未知响应
        raise RuntimeError(f"未知响应: code={code} status={status}")
    raise TimeoutError(f"{task_id} 客户端超时")
```

**推荐间隔**:
- **首 10s 内** 500ms 轮询一次(覆盖小文件命中缓存的快回)
- **10s ~ 60s** 1s/次
- **60s 之后** 2~5s/次(此时多半在 downloading 或解析大文件)

典型耗时:
- 命中缓存:1 次轮询(< 100ms)
- 小文件(< 5MB)冷启动:1-3s
- 大 PDF(50-200MB)冷启动:5-30s(下载 + 解析)
- 极大文件:5 分钟内完成,否则被看门狗 504 终止

## 4. 端到端完整示例 ⭐v2

**场景**:用户输入"中国哲学简史",展示列表,点某本后返回全文。
**关键变化**:加 429 退避重试 + 兼容 v2 新中间状态(pending/downloading/parsing)。

```python
import time
import random
import requests

BASE = "http://HOST:5555"
# 推荐默认过滤:4 种主流可解析格式 + 只要非小说类图书(见 3.1.1)
MAIN_EXTS = ["pdf", "epub", "fb2", "txt"]
CONTENT_TYPE = "book_nonfiction"

# v2:任何不是 completed/failed 的都属于"还在跑"
PENDING_STATES = {"pending", "downloading", "parsing", "processing"}
# 失败类错误码不重试,系统忙类错误码可重试
TRANSIENT_CODES = {429, 502}      # 临时:重试
PERMANENT_CODES = {404, 501, 503, 504}  # 永久:不重试


class ApiError(Exception):
    def __init__(self, code, message):
        super().__init__(f"[code={code}] {message}")
        self.code = code


def _unwrap(resp_json):
    """统一解包:code==0 返回 data;否则抛 ApiError"""
    if resp_json["code"] == 0:
        return resp_json["data"]
    raise ApiError(resp_json["code"], resp_json["message"])


def search(query: str) -> list[dict]:
    r = requests.get(f"{BASE}/api/search",
                     params={"q": query, "ext": MAIN_EXTS,
                             "content": CONTENT_TYPE},
                     timeout=10)
    data = _unwrap(r.json())
    return data["results"]


def submit_parse(md5: str, extension: str = "",
                 max_429_retries: int = 5) -> dict:
    """⭐v2 提交解析,自动处理 429 退避重试"""
    for attempt in range(max_429_retries):
        r = requests.post(f"{BASE}/api/parse",
                          json={"md5": md5, "extension": extension},
                          timeout=10)
        # 429:读 Retry-After 退避,加 jitter 防惊群
        if r.status_code == 429:
            wait = int(r.headers.get("Retry-After", "5"))
            wait += random.uniform(0, 1)
            time.sleep(wait)
            continue
        return _unwrap(r.json())
    raise ApiError(429, f"提交 {md5} 重试 {max_429_retries} 次仍 429")


def parse_book(md5: str, extension: str = "",
               poll_timeout: float = 600) -> str:
    """
    ⭐v2:
      - 提交端自动处理 429 退避
      - 轮询端兼容 pending/downloading/parsing 三种新中间状态
      - poll_timeout 默认 10 分钟(覆盖大文件下载+解析最坏情况)
    """
    task = submit_parse(md5, extension)
    task_id = task["task_id"]

    # 命中缓存直接拉文本
    if task["status"] == "completed":
        return _unwrap(
            requests.get(f"{BASE}/api/parse/{task_id}").json()
        )["text"]

    # 轮询
    t_end = time.time() + poll_timeout
    interval = 0.5  # 起步 500ms,逐步退避
    while time.time() < t_end:
        body = requests.get(f"{BASE}/api/parse/{task_id}").json()
        code = body["code"]
        data = body.get("data") or {}
        status = data.get("status")

        if code == 0 and status == "completed":
            return data["text"]
        if code in PERMANENT_CODES:
            raise ApiError(code, body["message"])
        # ⭐v2 兼容多个中间状态
        if code == 0 and status in PENDING_STATES:
            time.sleep(interval)
            interval = min(interval * 1.3, 5.0)  # 退避到 5s 上限
            continue
        # 502 重试一次(下载偶发失败)
        if code == 502:
            time.sleep(2)
            continue
        # 兜底
        raise ApiError(code, f"未知响应: status={status} {body['message']}")
    raise TimeoutError(f"解析 {md5} 客户端超时(>{poll_timeout}s)")


# 用法
results = search("中国哲学简史")
for b in results[:5]:
    print(b["md5"], b["title"][:30], b["extension"], b["filesize_str"])

text = parse_book(results[0]["md5"], results[0]["extension"])
print("全文长度:", len(text))
print(text[:200])
```

### 批量提交场景(并发) ⭐v2

```python
import concurrent.futures

def parse_many(books, max_workers=8):
    """
    并发解析多本书。
    上游 max_workers 建议 ≤ 服务端 PARSE_*_CONCURRENCY 总和的 1/2,
    避免直接打爆 429。
    """
    with concurrent.futures.ThreadPoolExecutor(max_workers=max_workers) as pool:
        futures = {
            pool.submit(parse_book, b["md5"], b["extension"]): b
            for b in books
        }
        for fut in concurrent.futures.as_completed(futures):
            book = futures[fut]
            try:
                text = fut.result()
                print(f"✅ {book['title'][:20]} → {len(text)} 字符")
            except ApiError as e:
                print(f"❌ {book['title'][:20]} → code={e.code}")
```

## 5. 行为说明(重要特性)

### 5.1 任务状态机 ⭐v2

```
                POST /api/parse
                     │
                     ▼
            Redis 已 completed?
              ┌──────┴──────┐
              │是           │否
              ▼             ▼
        cached=true      新建 meta
        立刻返回         status=pending
                            │
                            ▼
                  ┌─────────────────┐
                  │ 下载池 worker     │
                  │ status=downloading│
                  └────────┬────────┘
                           │
                ┌──────────┼──────────┐
                │下载失败    │下载成功
                ▼          ▼
          status=failed    按文件大小分流:
          code=502         ≤20MB → 小池(12 并发)
                          >20MB → 大池(4 并发)
                                  │
                                  ▼
                            status=parsing
                                  │
                       ┌──────────┴──────────┐
                       │成功                  │失败/超时
                       ▼                     ▼
                  status=completed       status=failed
                  写 Redis + 落盘         code=501/504
```

**客户端只需关心**:
- `status ∈ {pending, downloading, parsing, processing}` → 继续轮询
- `status == completed` → 拿 `text`
- `status == failed` → 看 `code` 决定怎么处理

### 5.2 解析请求去重 / 合流

- 同一 `md5 + engine` 在 10 分钟窗口内只会真正跑一次解析
- 10 个用户同时点同一本书 → 只会 1 次 CPU 解析,其他 9 个请求**合流**到同一 task_id
- 客户端对合流无感,轮询同一个 task_id 即可

### 5.3 不同书并行 + 大小文件分池 ⭐v2

服务端按文件大小路由到不同的线程池,**避免大文件挤占小文件的槽位**:

| 池 | 适用 | 默认并发(4C8G) | 调优 env |
|---|---|---|---|
| **下载池** | 所有冷启动任务 | 4 | `DOWNLOAD_CONCURRENCY=4` |
| **小文件解析池** | ≤ 20MB | 12 | `PARSE_SMALL_CONCURRENCY=12` |
| **大文件解析池** | > 20MB | 4 | `PARSE_LARGE_CONCURRENCY=4` |

大小分界由 `LARGE_FILE_THRESHOLD_MB`(默认 20)控制。

效果:
- 不会因为几个大 PDF 把所有解析槽位占满,小文件依然秒回
- 下载池独立,下载慢不阻塞已下载书的解析

### 5.4 过载背压(429)⭐v2

服务端实时跟踪两个计数:

```
queued    : 排队中(已 POST 还没起跑)
inflight  : 执行中(下载或解析中)
```

当 `queued >= max_queue_depth`(默认为各池容量总和 × 3 = 48),POST 立即返 **429 + Retry-After**。

**上游必须**:
- 看到 429 后**不立即重试**,读 `Retry-After` 头(秒)等待后再试
- 加 random jitter(0~1s)防惊群
- 上游自身的并发数不要太激进,建议 ≤ 服务端总并发的 1/2

### 5.5 看门狗任务超时(504)⭐v2

后台每 30s 扫描所有 in-flight 任务,**单任务超过 5 分钟**(env `TASK_TIMEOUT_SEC=300`)就强制标 failed,返回 `code=504`。

**作用**:防止 PyMuPDF 撞到奇葩 PDF 死循环、网络假死等"看似在跑实际卡死"的场景把槽位永久占住。

⚠️ 注意:看门狗能把 meta 改成 failed 让客户端立刻得到响应,**但已经卡住的 Python 线程无法被强行 kill**(语言限制),槽位要等线程自然返回才能回收。极端场景下可能小幅降低实际并发。

### 5.6 启动自愈 ⭐v2

服务进程重启时(重新部署、OOM 拉起等),自动执行:

1. 清掉所有遗留的 `parse:lock:*`(老进程的死锁)
2. 重置所有背压 counter 为 0(老进程的进程级状态已丢失)
3. 扫所有 `status=pending/downloading/parsing/processing` 的中间态 meta → **一律删除**,客户端下次 GET 会拿到 404,重提即可

**效果**:服务无状态,重启不会留下僵尸任务。重提成本很低 —— 如果原书还在 `books/` TTL 窗口内(生产 1h),二次解析不用重新下载;即使原书已被回收,只要 Redis 解析结果没过期(3h 内),命中后秒回。

### 5.7 解析结果的持久化(无状态设计)⭐

本服务是**中间解析能力**,不永久保存任何文件。**生产默认采用"省磁盘"策略**:

| 资源 | TTL | 理由 |
|---|---|---|
| **Redis 解析结果** | **3 小时**(`REDIS_PARSE_TTL=10800`) | 纯文本体积小,尽量保住,提升缓存命中率 |
| **原书缓存 `books/`** | **1 小时**(`BOOK_CACHE_TTL_SEC=3600`) | 原文件占磁盘大头(1 MB~500MB 不等),及时回收 |
| **上传临时文件 `uploads/`** | 解析结束立即删 | 纯中转 |

**命中路径**(同一本书在 3h 内重复请求):
- `1h 内`:Redis 命中,秒回 `completed`(~30ms)
- `1h ~ 3h`:Redis 命中,但原文件已删 —— 不影响!因为文本已在 Redis,不用再读原文件
- `3h 后`:Redis 过期 → 重新走搜索/下载/解析流程(端到端 5-10 秒)

> 代码默认值都是 `10800`(3h),生产通过 `docker-compose.yml` 的 env 覆盖 `BOOK_CACHE_TTL_SEC=3600` 改成 1h。如果业务特性需要全窗口命中,可自行改回 3h。

⭐v2 部署建议:Redis 开 AOF(`--appendonly yes`),避免重启丢 meta 导致窗口内请求被迫重解析。

### 5.8 上游镜像容灾

- 启动时从 6 个 Anna's Archive 官方镜像里并行探测、选最快
- 请求失败自动降级到备胎镜像
- 连续 3 次失败触发后台重新探测、重排镜像

### 5.9 搜索上游延迟

- 搜索直接爬 Anna's Archive HTML,**典型耗时 0.8 - 1.5s**
- 瓶颈在远端,服务本地处理 < 100ms

## 6. 错误码汇总 ⭐v2

统一用 `body.code` 判定,`HTTP` 仅参考。

| code | HTTP | 典型场景 | message 示例 | 上游处理 |
|---|---|---|---|---|
| `0` | 200 | 成功(含中间状态) | `"success"` | 完成 / 继续轮询 |
| `400` | 400 | 搜索缺 `q` / 解析缺 `md5` | `"请提供 md5..."` | 修参数,不重试 |
| `404` | 404 | md5 找不到文件 / task_id 已过期 | `"找不到文件..."` / `"任务不存在或已过期"` | 不重试,提示用户换 md5 |
| **`429`** ⭐ | 429 | 解析队列过载 | `"解析队列繁忙,请稍后重试"` | **读 Retry-After 退避重试** |
| `500` | 500 | 未预期服务端异常 | 内部报错消息 | 上报 + 慎重重试 |
| `501` | 200 | 解析失败(文件损坏、格式不支持) | `"不支持的格式..."` | 不重试,换 md5/engine |
| `502` | 502/200 | 上游不可用 / 下载失败 | `"所有镜像暂不可用..."` / `"下载失败..."` | 退避 30s 重试 |
| `503` | 200 | 解析完成但文本为空 | `"解析完成但文本内容为空..."` | 不重试,换 md5 |
| **`504`** ⭐ | 200 | 任务超时(>5 分钟) | `"任务超时(>300s),被看门狗终止"` | 不重试,换 md5 或上报 |

**判定建议(v2 完整版)**:

```python
import random, time

def handle_response(code, body, retry_count=0):
    if code == 0:
        return "ok"

    # 临时性失败,可重试
    if code == 429:
        # 读 Retry-After 退避 + jitter
        wait = int(body.get("retry_after", 5)) + random.uniform(0, 1)
        time.sleep(wait)
        return "retry"
    if code == 502:
        time.sleep(2 ** min(retry_count, 5))  # 指数退避
        return "retry"
    if code == 500:
        if retry_count < 2:
            time.sleep(5)
            return "retry"
        return "report_devs"

    # 永久性失败,不重试
    if code == 404:
        return "ask_user_change_md5"
    if code == 501 or code == 503 or code == 504:
        return "ask_user_change_md5"
    if code == 400:
        return "fix_request"

    return "report_devs"
```

## 7. FAQ

**Q: 搜到的 `md5` 一定能解析吗?**
A: 不一定。解析依赖能下载到原文件 —— 需要 `AA_SECRET_KEY`(Anna's Archive 会员密钥)或本地 `books/` 目录已经有该 md5 的文件。`/health` 返回 `download_api: "未配置"` 时后端下载不可用。

**Q: 同一本书用不同 engine 会冲突吗?**
A: 不会。`task_id = md5 + engine`,`pymupdf` 和 `pdftotext` 的结果各存各的,互不覆盖。

**Q: 文件超过多大解析会 OOM?**
A: PyMuPDF 对 500MB 以内 PDF 基本稳,EPUB/MOBI 按内存两倍估算。⭐v2 加了大文件单独低并发池(默认 4),超大文件被看门狗 5 分钟兜底,OOM 风险大幅降低。

**Q: 搜索结果里 `total` 为什么比 `results.length` 大?**
A: `total` 是上游标注的命中总数(如 443),`results` 只是当页(最多 50 条),用 `page` 参数翻页。

**Q: `cover_url` 前端加载失败?**
A: 加 `referrerpolicy="no-referrer"`:
```html
<img src="{cover_url}" referrerpolicy="no-referrer" />
```

**Q: 服务并发能到多少 QPS?** ⭐v2
A: 搜索受上游限制,单镜像约 10 QPS;解析并发 = 小池 + 大池(默认 4C8G 机器:12 + 4 = 16),超过自动排队;排队超过 `MAX_QUEUE_DEPTH`(默认 48)返 429。**4C8G 机器稳态吞吐 ≈ 8-15 本/秒**(混合分布),日处理量百万级。

**Q: 收到 429 怎么办?** ⭐v2
A: 读响应头 `Retry-After`(秒),等待后重试。**强烈建议加 0~1s random jitter** 防惊群。永远不要立即重试,否则上游一起冲会让服务雪崩。

**Q: 收到 504 怎么办?** ⭐v2
A: 该 md5 解析超过 5 分钟被看门狗强制终止,通常意味着文件损坏、格式异常、或文件太大。**不要重试**,引导用户换其他 md5。如果反复 504,可能是某个新格式触发了解析器死循环,需要后端排查。

**Q: POST 后立刻轮询,看到 `pending`/`downloading` 是正常的吗?** ⭐v2
A: 正常。v2 是真异步,POST 只登记,所以提交后马上轮询大概率看到 `pending`(还没被 worker 拾取)或 `downloading`(下载阶段)。客户端把任何不是 `completed`/`failed` 的状态都当"继续轮询"即可。

**Q: 服务重启后,我之前提交的任务还能拿到结果吗?** ⭐v2
A: 看情况:
- 任务**已经跑完且 Redis 未过期**(3h 内且 Redis 开了 AOF) → 下次 POST 命中缓存直接 `completed`
- 任务**还在中间态没跑完** → 重启时中间态 meta 被清理(服务无状态),客户端**重新 POST 即可**;如果原书还在 `books/` 窗口,二次解析不需要重新下载,速度很快

**Q: 为什么 3 小时后解析结果就拿不到了?** ⭐v2
A: 本服务定位是"中间解析能力",不做持久化存储,3h 窗口只为支撑上游重试。需要长期存储请在你的业务侧接住 `completed` 结果自行保存。如果业务特性需要更长窗口,可通过环境变量覆盖 `REDIS_PARSE_TTL` 和 `BOOK_CACHE_TTL_SEC`。
